Antarctic Expedition Agent

Antartia

An autonomous AI field agent that tracks an Antarctic expedition in real time — GPS, weather, photos, and knowledge — driven entirely by a local LLM through recursive tool chaining.

Python 3.11+ asyncio Ollama qwen2.5vl:7b ChromaDB SQLite

Architecture

Everything runs on a single asyncio event loop. The LLM decides what to do through recursive tool chaining — it emits actions, the runtime executes them and appends results to context, then the LLM is re-invoked. This repeats until finish.

Recursive Tool Chaining

LLM emits actions → tools execute → results appended to context → LLM re-invoked. Up to max_chain_depth = 6. No hardcoded orchestration.

Execution Semaphore

Single asyncio.Lock coordinates CLI, scheduler, and task runner. Four states: idle → user_typing → llm_running → idle. HTTP server never touches it.

All-Local Inference

Ollama serves all models locally. qwen2.5vl:7b for vision, text model for chat and scoring, nomic-embed-text for embeddings. No GPU required.

DB-Backed Tasks

Tasks persist in SQLite (tasks table). FIFO ordering. Survive restarts. Scheduler claims the oldest pending task each 60s tick.

HTTP GPS Ingestion

POST /locations from iPhone Shortcut. Asyncio server, no framework. Never acquires the semaphore — GPS always recorded immediately.

Protocol-Based

LLMClient, StateStore, OutputHandler are Python Protocols. Ollama, OpenRouter, memory store, file store — all swappable.

Execution Semaphore

A single asyncio.Lock with explicit state labels governs every heavy operation. The CLI acquires the lock before showing the ❯ prompt and holds it through the entire agent reply. The scheduler gets at most one window between turns.

While a background task runs, the CLI waits at acquire_typing() — the ❯ prompt is blocked and a spinner is shown. Task progress streams to the scroll area via on_task_progress().

Photo Pipeline

Photos dropped into data/photos/inbox/ are processed inline when the LLM calls scan_photo_inbox, or asynchronously when the scheduler picks up a process_photo task.

inbox/photo.jpg → ImagePreprocessingService EXIF correction · resize 640–800px · SHA-256
vision_preview/photo_preview.jpg → OllamaVisionClient qwen2.5vl:7b · base64 → {description, summary}
description → _score_significance() Ollama text call → significance_score 0.0–1.0
score ≥ 0.75? → is_remote_candidate = true eligible for upload to expedition website
original → processed/photo.jpg moved, never modified

The significance scoring prompt is configurable via photo_pipeline.scoring_prompt in the expedition config — no code changes needed to tune the threshold criteria.

Knowledge Base

A local ChromaDB instance stores semantic embeddings of expedition documents — itinerary, species guides, ship specs, location context, science notes. The LLM can query it via search_knowledge before answering field questions.

data/knowledge/*.txt|*.md → chunk ~500 chars → Ollama /api/embed → ChromaDB upsert
search_knowledge query → embed query → top-5 chunks → tool result → LLM context

No server required. ChromaDB runs embedded and persists to data/knowledge_db/. Embedding model: nomic-embed-text (274 MB) via the same Ollama instance already running for vision and chat.

Key Design Decisions

  • finish is the only terminal action send_message displays text but does not stop the chain. The LLM can emit multiple send_message calls (progress updates) alongside tool actions in one turn. Only finish stops the loop and releases the semaphore.
  • Semaphore holds through the full user interaction cycle The CLI acquires the lock before showing ❯. It stays held through typing, LLM calls, and all tool executions — released only after finish. This guarantees the scheduler can never interrupt a conversation mid-chain.
  • HTTP server is lock-free by design GPS coordinates from the iPhone are always stored immediately, regardless of what the CLI or scheduler is doing. Task queue accumulates freely; execution is deferred until the lock is free.
  • All paths from environment variables — no defaults in code Every file path, URL, and port is read from os.environ["KEY"]. Pydantic config fields use Field(default_factory=lambda: os.environ["KEY"]). Missing env var = hard crash at startup, not a silent default.
  • Ollama for all inference — no cloud dependency Vision (qwen2.5vl:7b), significance scoring, embeddings (nomic-embed-text) all via the local Ollama server. The agent works fully offline in the field.
  • Significance scoring prompt is configurable, not hardcoded photo_pipeline.scoring_prompt in the JSON config controls what the LLM considers significant. Threshold (0.75) is also configurable. No code changes needed to tune field-specific criteria.
  • Original photos are never modified ImagePreprocessingService creates a derived JPEG preview. The original moves from inbox/ to processed/ after successful analysis. Preview files are disposable.
  • Repository pattern — one class per SQLite table Each repo takes a Database reference and issues raw SQL. No ORM. WAL mode enabled. Repos have no knowledge of each other.
